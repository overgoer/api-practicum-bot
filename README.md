# API Practicum Bot — Telegram-бот для воронки продаж

**API Practicum Bot** — Telegram-бот, который проводит пользователей через мини-практикум по поиску багов в API. Четыре задачи, разбор ошибок и прогрев к покупке полного курса [API Practicum](https://t.me/api_practikum_bot).

Бот — первая ступень продуктовой воронки: бесплатные задачи демонстрируют ценность навыка, а после прохождения пользователь получает оффер на платный доступ к стенду с 15+ багами.

## 🐛 Задачи мини-практикума

### Задача 1: Create Order — `success: true` при ошибке

**Эндпоинт:** `POST /api/v3/orders/create`

Тело запроса — `item_id: 404`, `quantity: 0`, `promo_code: "WINTER2026"`. Сервер возвращает `200 OK` с полями:

```json
{
  "success": true,
  "order_id": null,
  "message": "Quantity must be more than 0"
}
```

**Баг:** поле `success: true` противоречит сообщению об ошибке и отсутствию `order_id`. Клиент не может понять — заказ создан или нет.

**Правильный ответ:** противоречие в теле ответа (`success: true` при ошибке).

---

### Задача 2: Profile — утечка `password_hash`

**Эндпоинт:** `GET /api/v2/profile?user_id={userId}` с заголовком `X-Auth-Token: admin_token_99`

Сервер возвращает профиль с полем:

```json
{
  "password_hash": "sha256$uI89!jd#k9",
  ...
}
```

**Баг (security):** хеш пароля никогда не должен возвращаться в API-ответе. Утечка даёт злоумышленнику возможность брутфорс-атаки, pass-the-hash и обхода аутентификации.

**Правильный ответ:** `password_hash` в ответе — критическая уязвимость.

---

### Задача 3: Products — фильтры не работают

**Эндпоинт:** `GET /api/products?status=active&has_discount=true&limit=2`

В ответе приходит товар со статусом `"archived"` и без скидки (`price == old_price`).

**Баг:** оба фильтра (`status`, `has_discount`) не применяются на стороне бэкенда — сервер возвращает что угодно.

**Правильный ответ:** фильтры игнорируются, API отдаёт мусор.

---

### Задача 4: Cart Item — отрицательное количество

**Эндпоинт:** `PATCH /api/cart/items/55` с телом `{"quantity": -1}`

Сервер возвращает:

```json
{
  "new_quantity": -1,
  "total_price": -499.00
}
```

**Баг:** сервер принимает отрицательное количество и считает отрицательную цену. Это позволяет манипулировать итоговой суммой заказа, ломать отчётность и аналитику.

**Правильный ответ:** и `quantity`, и `total_price` — баги. Должна быть валидация `quantity <= 0 → 400 Bad Request`.

## 📈 Воронка продаж

Бот реализует полную последовательность прогрева к покупке:

1. **Задачи (4 шт.)** — пользователь ищет баги, получает разбор. Демонстрация ценности навыка.
2. **История из продакшена** — рассказ про баг с пробелом в статусе заказа (`"NEW "` vs `"NEW"`), который «убил» 100% заказов на проде.
3. **Выбор боли** — пользователь выбирает, какая проблема тестирования API ему ближе (200 OK с ошибкой в теле, путаница в авторизации, оформление баг-репорта). Выбор сохраняется в БД для персонализации.
4. **Показ оффера** — описание API Practicum: стенд на 30 дней, 15+ багов, Postman-коллекция, видео-разборы.
5. **Принятие / отказ** — при согласии пользователь перенаправляется в личку @eddytester для оплаты. При отказе — получает 3 полезных материала и возможность вернуться через `/start`.

## 🏗 Архитектура

```
┌─────────────┐     Telegram API     ┌───────────────────┐
│  Пользователь │ ◄────────────────── │  api-practicum-bot │
│  Telegram     │                     │  (aiogram 3.x)     │
└─────────────┘                      └────────┬──────────┘
                                              │
                                     ┌────────▼──────────┐
                                     │  SQLite (practicum.db) │
                                     │  • users           │
                                     │  • step / pain     │
                                     │  • offer_status    │
                                     └───────────────────┘
```

- **Фреймворк:** [aiogram 3.x](https://docs.aiogram.dev/) — асинхронный фреймворк для Telegram Bot API
- **База данных:** SQLite3 — файл `practicum.db` (или путь из переменной `DB_PATH`)
- **FSM:** встроенный `MemoryStorage` для управления диалогом
- **Парсинг:** HTML-разметка в сообщениях (`parse_mode="HTML"`)

### Директория `balance_lab/`

Отдельный Docker-стенд для демонстрации проблем с балансировкой и заголовком `Host`. Три бэкенда (`api`, `app`, `admin`) за Nginx. Каждый бэкенд — простой HTTP-сервер на Python, который возвращает информацию о запросе: `Host`, `X-Forwarded-For`, все заголовки. Используется как демо-материал для задачи про Host-заголовок.

## 🚀 Установка

### Требования

- Python 3.10+
- pip

### 1. Клонирование

```bash
git clone <url-репозитория>
cd api-practicum-bot
```

### 2. Зависимости

```bash
pip install -r requirements.txt
```

Файл `requirements.txt`:

```
aiogram==3.27.0
aiohttp==3.13.5
aiofiles==25.1.0
```

### 3. Настройка

Создайте файл `.env` или экспортируйте переменные:

```bash
export BOT_TOKEN="ваш_токен_от_@BotFather"
export DB_PATH="/path/to/practicum.db"
```

> ⚠️ **Важно:** Токен в коде (`TOKEN` на строке 25 `main.py`) имеет значение по умолчанию для разработки. Для продакшена **обязательно** задавайте `BOT_TOKEN` через переменную окружения.

### 4. Запуск

```bash
python main.py
```

Или через Docker:

```bash
docker build -t api-practicum-bot .
docker run -d \
  -e BOT_TOKEN="ваш_токен" \
  -e DB_PATH=/data/practicum.db \
  -v ./data:/data \
  api-practicum-bot
```

## 🔧 Переменные окружения

| Переменная     | По умолчанию        | Описание                                      |
| -------------- | ------------------- | --------------------------------------------- |
| `BOT_TOKEN`    | хардкод в `main.py` | Токен Telegram-бота от @BotFather             |
| `DB_PATH`      | `practicum.db`      | Путь к файлу SQLite3                          |

> [ISSUE] Токен по умолчанию жёстко зашит в `main.py` (строка 25). В продакшене токен должен задаваться **только** через переменную окружения, без fallback-значения в коде.
